# 家計簿アプリ仕様書

## 1. プロダクトの目的と設計思想

### プロダクトの目的
家族（2名）の共有支出を可視化し、無駄な出費を抑えつつ楽しく貯金することを目的とします。

### 設計思想
アプリの使い勝手よりも、運用方法の設計とエッジケースへの耐性を最優先します。非機能要件については「不満が溜まったら改善する」方針とし、初期段階での過剰な作り込みは行いません。データの信頼性を担保するため、システムへの入力はLINE（AI解析）およびスプレッドシートへの直接編集に限定し、UIは閲覧専用のダッシュボードとして機能させます。

## 2. 対象外要件（Out of Scope）

以下の機能および要件は、本システムでは実装・対応しません。
- 銀行APIやクレジットカードAPIを利用した明細の自動連携
- ダッシュボードUI上でのデータ登録、編集、削除（入金登録を含む）
- 個人の給与口座からの支出や、個人的な買い物の管理
- 高度な名寄せ処理（AIによる店舗名とカード明細の請求元名の完全一致判定など）
- 自動化された月次バッチ処理（当面はスプレッドシートの手動運用とする）

## 3. ユビキタス言語

- 元帳（明細シート）：すべての入出金履歴が記録されるトランザクションデータ。
- 財布：予算と資金を管理する独立したバケツ（サブ口座）。月次の生活費（例：2026-05通常）やイベント（例：沖縄旅行）など、それぞれを独立した一意の名前で管理します。
- カテゴリ：支出の分類。食費や日用品費など、分析の最小単位。
- 予算記録：特定の財布、カテゴリに対して設定される目標金額の記録。
- 資金移動（振り替え）：ある財布から別の財布へ資金を移すこと。元帳における支出レコードと入金レコードのペアとして表現されます。
- 立て替え相殺：個人が支払った額を、翌月の固定入金額から減額して精算する仕組み。
- 赤黒処理：キャンセル発生時に、過去データを修正せずマイナスデータを新規登録する処理。

## 4. アクターと認証認可

### アクターの定義
- ユーザーA / ユーザーB：家計簿を利用する家族。
- LINE AI：メッセージから意図を解釈し、スプレッドシートへデータを書き込むインターフェース。
- 管理システム：LIFF SDKを介してユーザーを識別し、閲覧権限の制御とデータの集計を行います。

### 認証・認可の仕組み
- ユーザー識別：LINEユーザーIDを鍵としてユーザーを識別します。
- 認証：LINEアカウントを認証基盤とし、パスワード管理は行いません。
- LIFF（LINE Front-end Framework）の活用：ダッシュボードをLIFFアプリとして実装し、トーク画面からシームレスにアクセスし、自動的にユーザーIDを取得します。
- アクセス制限：事前にユーザーマスタに登録された特定のID以外からのアクセスは遮断します。

## 5. システム構成とデータフロー

### 入力フロー
1. ユーザーがLINEへ画像またはテキストを送信します。
2. LINE AIが金額・日時・カテゴリ・財布・アクターを特定します。
3. AIがスプレッドシートの元帳へレコードを追記します。
4. 財布マスタや予算記録の更新は、ユーザーがスプレッドシートを直接編集します。

### 出力フロー
1. ユーザーがLINEのリッチメニューからダッシュボードを開きます。
2. システムがLIFF経由でアクセス者のLINEユーザーIDを照合します。
3. 元帳およびマスタデータを読み込み、集計結果を画面に描画します。

## 7. ダッシュボード表示ロジック

ダッシュボードは単一画面で構成され、以下の優先順位で情報を表示します。

### 当月の通常財布セクション
- 判定：現在日時（YYYY-MM）から「YYYY-MM通常」という名称の財布を検索。
- 表示：全体の消化率（プログレスバー）およびカテゴリ別の消化状況。
- 警告：該当する財布名がマスタに存在しない場合、「今月の通常財布が未設定です。スプレッドシートで作成してください」という警告を表示。

### 特別財布セクション
- 判定：財布タイプが「一括」となっているアクティブな財布を抽出。
- 表示：財布ごとの予算に対する累計支出のプログレスバーをカード形式で羅列。
- 備考：完了した財布は、スプレッドシート側で物理的に移動させるか非表示にする運用で対応。

### 精算タスクセクション
- 判定：閲覧ユーザーの当月の立替額を集計。
- 表示：翌月の入金要件額（固定基本額との相殺結果）を表示。

## 8. 計算ロジック

- 翌月入金額の計算：
  個人の翌月入金額 ＝ 固定基本額 －（個人の通常立替支出合計）＋（個人が負担すべき特別財布の立替支出合計）
  ※精算フラグがTRUEのレコードのみを対象とします。
- 端数処理：
  特別財布の折半で発生する端数は、決済日時の月に応じて負担者を交代（偶数月：A負担、奇数月：B負担）。

## 9. 例外フローとエラーハンドリング

- データのリカバリ：スプレッドシートの標準機能である変更履歴を活用。
- 解析失敗：AIが解析に失敗した場合は、LINEを通じて通知し、手動修正を促します。
