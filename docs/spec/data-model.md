# データモデル

スプレッドシート上に存在するシートと列の定義。**過去の決定として確定した構造** を記録する。型定義の正は `app/domain/` の valibot スキーマ。

## リレーション方針

すべてのリレーションは ID ではなく **「文字列のマッチング」** で表現する。財布名・カテゴリ名・アクター名は人間が読める一意な文字列とし、外部キー的な役割を兼ねる。

理由: スプレッドシート上で人間が直接編集・確認することを前提とする運用のため、ID による参照だと可読性が著しく落ちる。文字列の表記揺れによるマッチング漏れは「マスタ未登録扱い」として明示的に検知する設計で許容する。

| :--- | :--- |

### 財布マスタ

| カラム名 | 内容 | 備考 |
| :--- | :--- | :--- |
| 財布名 | 一意な名称 | 例: `2026-05通常`, `沖縄旅行` |
| 財布タイプ | `月次` または `一括` | 表示ロジックの切り替え用 |

#### 財布名の規約

- **通常財布**: `YYYY-MM通常` 形式で、システムが機械生成する。AI が返却した取引日の年月から自動的に組み立てる（実装: `app/features/ledger/register.ts` の `toLedgerEntry`）。
- **特別財布**: 人間がスプレッドシート上で任意の名前を付与する（例: `沖縄旅行`）。

決定の理由: 通常財布名を機械生成にすることで、AI 側に「今月の財布名は何か」を推測させる必要がなくなり、入力時点での財布マスタ参照が不要になる。月をまたぐ取引や月末取引でも、AI が返す `date` の年月から決定論的に財布が決まる。

### カテゴリマスタ

| :--- | :--- |
| カテゴリ名 | 食費、日用品など |

## トランザクション系データ

### 元帳（入出金明細）

| カラム名 | 内容 | データ型 | 備考 |
| :--- | :--- | :--- | :--- |
| トランザクション ID | 一意な識別子 | 文字列 | |
| 日時 | 取引日時 | 日付時刻 | |
| 入出金区分 | `入金` または `支出` | 文字列 | |
| 金額 | 取引額 | 数値 | |
| アクター | 支払・操作主体 | 文字列 | 元帳上は常に `共同` で統一する（後述） |
| カテゴリ | 紐付くカテゴリ名 | 文字列 | カテゴリマスタと一致させる |
| 財布 | 紐付く財布名 | 文字列 | 財布マスタと一致させる |
| 精算フラグ | 自動精算対象か | 真偽値 | デフォルト TRUE |

#### アクターを `共同` に統一する決定

元帳に書き込む `actor` は、認証済みであれば常に `共同` に統一する（実装: `app/features/ledger/register.ts`）。個人立替の判定は元帳ではなく別シート（または運用ルール）で行う、という二層構造を採る。

理由: 「個人の立替」と「共同支出」を元帳の同じ列で混在管理すると、精算ロジックの引数が膨らみ、ダッシュボード集計のクエリが複雑化する。元帳は「家計に属する取引のフラットなログ」として単純化する。

## 3. 入出金エントリの型分離

### 設計方針

元帳シートには入金・支出を同一行フォーマットで記録する（DB・集計の都合）。しかしドメインとしては **入金と支出は異なる概念** であり、持つべきフィールドが異なる。この差異をレイヤーで分離する。

| レイヤ | 型 | 説明 |
|-------|----|----|
| domain | `IncomeEntry` / `SpendingEntry` | 概念として正しい型。features 層はこちらを使う |
| infra / storage | `LedgerRow` | シートへの書き込み・読み出しフォーマット。`type` フィールドで discriminate する |

### IncomeEntry（入金）

```
date, amount, actor, memo
```

入金は「口座にお金が入ってきた」という事実のみを記録する。財布・カテゴリ・精算フラグはドメインとして存在しない。

**財布を持たない理由**: 月別節約額は `(予算 − 支出)` であり入金の財布に依存しない。推定残高は全入金の合計であり財布による分類が不要。入金の用途はさまざまで（毎月の生活費、クレカ補填、特別財布の精算など）すべてを財布に分類しようとすると実態に合わないモデルになる。

### SpendingEntry（支出）

```
date, amount, actor, memo, wallet, category, shouldSettle
```

支出は「どの財布の、どのカテゴリに対する出費か」を持つ。予算との照合・貯金計算・精算計算はすべてこのフィールドに依存する。

### LedgerRow（ストレージフォーマット）

```
type("入金" | "支出") + 上記の全フィールドを持つ flat な行
```

シートには入金・支出を同一行として書き込む。SpendingEntry にない入金側のフィールド（wallet・category・shouldSettle）はシート上で空値になる。infra 層が `type` フィールドで判別し、適切なドメイン型に変換して features 層へ渡す。

### AI パーサー出力

AI が返す解析結果も同様に分離する。

| 型 | フィールド |
|----|---------|
| `ParsedIncome` | `type:"入金"`, `date?`, `amount`, `memo` |
| `ParsedSpending` | `type:"支出"`, `date?`, `amount`, `category`, `memo` |

入金時に AI が財布やカテゴリを推測する必要がなくなり、プロンプトを単純化できる。`wallet` は features 層が `date` から機械的に組み立てる（支出のみ）。
