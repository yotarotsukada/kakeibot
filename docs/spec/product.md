# プロダクト仕様

本ドキュメントは kakeibot の目的・スコープ・アクター・認証認可・画面構成といった「**過去の決定として確定したプロダクト要件**」を保管する。実装の根拠を確認するときに参照する。

ドメイン用語そのものの定義は `.claude/skills/domain-knowledge/` を参照。計算式の詳細は `./calculation.md`、データ構造は `./data-model.md` を参照。

---

## 1. プロダクトの目的と設計思想

### プロダクトの目的

家族（2 名）の共有支出を可視化し、無駄な出費を抑えつつ楽しく貯金することを目的とする。

### 設計思想

アプリの使い勝手よりも、運用方法の設計とエッジケースへの耐性を最優先する。非機能要件については「不満が溜まったら改善する」方針とし、初期段階での過剰な作り込みは行わない。

データの信頼性を担保するため、システムへの入力は **LINE（AI 解析）およびスプレッドシートへの直接編集に限定** し、UI は閲覧専用のダッシュボードとして機能させる。

## 2. 対象外要件（Out of Scope）

以下の機能および要件は本システムでは実装・対応しない。

- 銀行 API やクレジットカード API を利用した明細の自動連携
- ダッシュボード UI 上でのデータ登録、編集、削除（入金登録を含む）
- 個人の給与口座からの支出や、個人的な買い物の管理
- 高度な名寄せ処理（AI による店舗名とカード明細の請求元名の完全一致判定など）
- 自動化された月次バッチ処理（当面はスプレッドシートの手動運用とする）

## 3. アクターと認証認可

### アクターの定義

- **ユーザー A / ユーザー B**: 家計簿を利用する家族。
- **LINE AI**: メッセージから意図を解釈し、スプレッドシートへデータを書き込むインターフェース。
- **管理システム**: LIFF SDK を介してユーザーを識別し、閲覧権限の制御とデータの集計を行う。

### 認証・認可の仕組み

- **ユーザー識別**: LINE ユーザー ID を鍵としてユーザーを識別する。
- **認証**: LINE アカウントを認証基盤とし、パスワード管理は行わない。
- **LIFF（LINE Front-end Framework）の活用**: ダッシュボードを LIFF アプリとして実装し、トーク画面からシームレスにアクセスして自動的にユーザー ID を取得する。
- **アクセス制限**: 事前にユーザーマスタに登録された特定の ID 以外からのアクセスは遮断する。

## 4. システム構成とデータフロー

### 入力フロー

1. ユーザーが LINE へ画像またはテキストを送信する。
2. LINE AI が金額・日時・カテゴリ・財布・アクターを特定する。
3. AI がスプレッドシートの元帳へレコードを追記する。
4. 財布マスタや予算記録の更新は、ユーザーがスプレッドシートを直接編集する。

### 出力フロー

1. ユーザーが LINE のリッチメニューからダッシュボードを開く。
2. システムが LIFF 経由でアクセス者の LINE ユーザー ID を照合する。
3. 元帳およびマスタデータを読み込み、集計結果を画面に描画する。

## 5. ダッシュボード表示ロジック

ダッシュボードは単一画面で構成され、以下の優先順位で情報を表示する。

### 当月の通常財布セクション

- **判定**: 現在日時（`YYYY-MM`）から「`YYYY-MM通常`」という名称の財布を検索。
- **表示**: 全体の消化率（プログレスバー）およびカテゴリ別の消化状況。
- **警告**: 該当する財布名がマスタに存在しない場合、「今月の通常財布が未設定です。スプレッドシートで作成してください」という警告を表示。

### 特別財布セクション

- **判定**: 財布タイプが「一括」となっているアクティブな財布を抽出。
- **表示**: 財布ごとの予算に対する累計支出のプログレスバーをカード形式で羅列。
- **備考**: 完了した財布は、スプレッドシート側で物理的に移動させるか非表示にする運用で対応。

### 精算タスクセクション

- **判定**: 閲覧ユーザーの当月の立替額を集計。
- **表示**: 翌月の入金要件額（固定基本額との相殺結果）を表示。

## 6. 例外フローとエラーハンドリングの方針

- **データのリカバリ**: スプレッドシートの標準機能である変更履歴を活用する。
- **解析失敗**: AI が解析に失敗した場合は、LINE を通じて通知し、手動修正を促す。
- **アプリ内エラー表現**: 詳細は `app/domain/errors.ts` の `AppError` 階層を参照。
