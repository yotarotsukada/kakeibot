# 貯金・残高機能 仕様

本ドキュメントは `/savings` 画面の仕様・概念定義・設計上の決定とその背景を記録する。

関連ドキュメント: `docs/spec/data-model.md`（入出金エントリの型分離）、`docs/spec/calculation.md`（月次精算計算）

---

## 1. ユースケースと設計の目的

この機能が答えたい問いは3つ。

| 問い | 概念 |
|------|------|
| 今の共同口座にいくらあるか（目安） | **推定残高** |
| 月ごとにどれくらい節約できているか、その推移は | **月別節約額** |
| これまでの累計でいくら貯まったか | **累計貯金額** |

### 提供しない情報

以下は意図的にこの画面に含めない。

- 予算の前月比較（`/home` の通常財布カードで把握できる）
- 特定期間を選んだ貯金額の集計（期間フィルタは不要。常に「全期間累計」と「直近6ヶ月グラフ」で十分）

---

## 2. 概念定義

### 推定残高

**定義**: 元帳に記録された全入金の合計 − 全支出の合計（全財布・全期間）

kakeibot の使い始めを起算点とする。口座の初期残高がある場合は「入金」として元帳に登録する運用で対応する。

「推定」と呼ぶのは、クレカ引き落としのタイミングを考慮しないため。支出は購入時点（実際の引き落とし前）に元帳へ記録されるため、実際の口座残高とはずれる。精度よりも「目安としての即答性」を優先する設計判断。

### 月別節約額

**定義**: その月の通常財布の予算合計 − 通常財布の支出合計

- 通常財布（`YYYY-MM通常`）のみを対象とする
- 正の値 = 節約（予算内）、負の値 = 予算超過

### 累計貯金額

**定義**: 月別節約額の全期間累計 = Σ（通常財布の月別節約額）

こちらも通常財布のみ。「毎月の生活費の中でいくら節約してきたか」の積み上げ。

---

## 3. 特別財布を貯金計算から除外する理由

特別財布は **貯金の使い道** であって **貯金の源泉** ではない。

月々の通常財布の節約で貯まったお金が、特別財布の目標（旅行・家具など）へ充てられる、という資金フローが実態に近い。そのため「月々いくら節約できたか」の指標には特別財布を含めない。

特別財布を含めた場合の具体的な問題:

- 特別財布の支出は複数月にまたがるため、月次の節約評価になじまない
- 予算未設定の特別財布の支出が、そのまま累計貯金額をマイナス方向に引っ張る
- 特別財布の精算（入金）は手動登録であり、支出だけが先に計上される期間が生まれる

なお、**推定残高**については特別財布の支出・入金も含めて全エントリを集計する。特別財布の精算が完了すれば対応する入金が登録されるため、口座残高の推移として正しく反映される。

---

## 4. 入金と財布の関係

入金エントリには財布の概念が不要。理由:

- 月別節約額は `(予算 − 支出)` であり、入金の額や財布には依存しない
- 推定残高は全入金の合計であり、財布による分類は必要ない

入金の用途はさまざまであり（毎月の生活費、クレカ補填のための入金、特別財布の精算入金など）、それらをすべて財布に紐づけようとすると実態に合わないモデルになる。

この結論は `docs/spec/data-model.md` のエントリ型分離の設計につながる。

## 5. /savings 画面仕様

### 5.1 表示セクション

**① 推定残高と累計貯金額（ヒーロー）**

最も目立つ場所に2つを並べて表示する。

| 表示 | 定義 |
|------|------|
| 推定残高 | Σ全入金 − Σ全支出（全期間） |
| 累計貯金額 | Σ（通常財布の月別節約額）全期間 |

「口座にいくらある（残高）」と「節約でいくら貯めた（貯金）」は答えが異なる。両者を並べることで家計の実態をより正確に伝える。

**② 月別節約額グラフ**

- 直近6ヶ月の棒グラフ
- 正の棒（節約月）と負の棒（予算超過月）を色で区別
- X軸: 月名、Y軸: 金額

**③ 月別内訳リスト**

各月のカード形式で以下を表示（新しい月が上）:

| 項目 | 内容 |
|------|------|
| 入金 | その月の全入金合計 |
| 通常支出 | その月の通常財布の支出合計 |
| 予算 | その月の通常財布の予算合計 |
| 節約額 | 予算 − 通常支出（バッジ表示） |

### 5.2 表示しない情報

- 入金の財布別内訳（入金に財布の概念は不要）
- 特別財布の月別支出内訳

## 6. 設計経緯と引き継ぎ事項

### 6.1 当初実装で判明した問題点

最初の実装では `savedAmount` の計算に `getLedgerEntriesByMonth`（全財布・月別）を使っていたため、特別財布の支出が混入していた。

```
savedAmount = totalBudget(通常財布) − totalSpending(全財布) ← 誤り
```

正しくは通常財布の支出のみで計算する必要がある。

```
savedAmount = totalBudget(通常財布) − normalWalletSpending(通常財布のみ) ← 正しい
```

月別内訳の表示においても、`totalSpending`（全財布）と `totalBudget`（通常財布のみ）を同じ行に並べると、比較の基準が合わず誤解を招く。

### 6.2 「財布に紐づかない入金」ユースケースの発見

実装・議論の過程で、クレカ補填など財布に紐づかない入金が実際の運用に存在することが判明した。これが「入金に財布は不要」という結論につながった（§4 参照）。

### 6.3 累計貯金額の定義変更

当初は「全財布の予算 − 全財布の支出」（特別財布含む）を累計貯金額として定義していた。しかし特別財布は「貯金の使い道」であるという整理から、通常財布の月別節約額の累計のみを累計貯金額と定義し直した。

### 6.4 データモデルの分離

元帳シートへの記録は入金・支出を同一フォーマットで扱う（DB都合）が、ドメインとしては異なる概念であることを明確化した。この決定の詳細は `docs/spec/data-model.md` §3 を参照。
