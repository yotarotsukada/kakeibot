# 計算ロジックと精算ルール

家計簿アプリのコア計算ロジックを「**決定 / 背景 / 式**」の組で記述する。実装の根拠を確認したいときに参照する。実装本体は `app/domain/budget/` および `app/features/budget/`。

---

## 1. 翌月入金額の計算

### 決定

各個人の翌月入金額は、固定基本額から個人の通常財布での立替額を差し引き、特別財布の負担分を加算して求める。

```
個人の翌月入金額 = 固定基本額 −（個人の通常立替支出合計）+（個人が負担すべき特別財布の立替支出合計）
```

精算フラグ（`shouldSettle`）が TRUE のレコードのみを対象とする。

### 背景

家族間で月次の現金移動を最小化するため、毎月の入金額を立替額に応じて自動調整する。これにより「個別に精算する」運用を省き、固定スケジュールでの 1 回の入金に集約できる。

---

## 2. 精算フラグのデフォルト値

### 決定

新規取引レコードを登録する際、`shouldSettle` のデフォルトは **常に true** とする（実装: `app/features/ledger/register.ts`）。

### 背景

家族の共有家計簿という用途上、**ほぼすべての取引が精算対象** になる。「精算しない取引（プレゼントを贈ったときの個人負担など）」は例外であり、その都度フラグを倒す運用としたほうがミスが少ない。

明示的に精算対象外にしたい場合は、スプレッドシート上で当該レコードのフラグを後から FALSE に書き換える。

---

## 3. アクターの統一と立替判定の二層化

### 決定

元帳に書き込むアクター列は、認証済みであれば常に文字列 `共同` に統一する。個人がどれだけ立替えたかは、元帳ではなく **別シートまたは運用ルール** で管理する。

### 背景

元帳を「家計に属する取引のフラットなログ」として単純化することで、ダッシュボード集計のクエリと AI のプロンプトを単純に保つ。AI に「立替か共同か」を判断させる必要がなくなる。

詳細は `./data-model.md` の元帳セクションを参照。

---

## 4. 特別財布の端数処理

### 決定

特別財布における折半で 1 円未満の端数が発生する場合、決済日時の **月に応じて負担者を交代** する。

- 偶数月（2, 4, 6, ...）: A が端数を負担
- 奇数月（1, 3, 5, ...）: B が端数を負担

### 背景

長期で見たときに不公平が累積しないようにする。月次でローテーションする方式は、計算ロジックが決済日時 1 つから決定論的に求まるため、状態を別管理せずに済む。

---

## 5. 赤黒処理（キャンセル）

### 決定

キャンセルが発生した場合、過去レコードを修正せず **マイナス金額の新規レコードを追加** する。

### 背景

スプレッドシートを直接編集する運用上、過去レコードの修正は変更履歴を辿りにくく、集計途中のスナップショットと整合が取れなくなる。キャンセルを赤黒で表現することで、元帳は append-only に保たれる。
