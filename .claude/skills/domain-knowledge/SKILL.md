## 1. ユビキタス言語

| 用語 | 定義 |
|------|-----|
| **元帳**（明細シート） | すべての入出金履歴が記録されるトランザクションデータ。append-only。 |
| **財布** | 予算と資金を管理する独立したバケツ（サブ口座）。月次の生活費や旅行などのイベントを、それぞれ独立した一意の名前で管理する。 |
| **通常財布** | 月次の生活費を入れる財布。`YYYY-MM通常` の機械生成名を持つ。 |
| **特別財布** | 旅行など、月をまたぐ目標に紐付く財布。人間が任意の名前を付ける。 |
| **カテゴリ** | 支出の分類。食費・日用品など、分析の最小単位。 |
| **予算記録** | 特定の財布 × カテゴリに対して設定する目標金額。 |
| **資金移動**（振り替え） | ある財布から別の財布へ資金を移すこと。元帳上では支出レコードと入金レコードのペアで表現する。 |
| **立替相殺** | 個人が支払った額を、翌月の固定入金額から減額して精算する仕組み。 |
| **赤黒処理** | キャンセル発生時に、過去データを修正せずマイナスデータを新規登録する処理。 |
| **アクター** | 取引の操作主体。マスタ上は A, B など個人名で持つが、元帳上は常に `共同` に統一する（後述）。 |
| **精算フラグ**（`shouldSettle`） | 立替相殺の対象に含めるかどうかの真偽値。 |

### よくある誤用

- 「財布」と「口座」を混同しない。kakeibot の財布は **論理的な予算バケツ** であり、実在の銀行口座とは無関係。
- 「カテゴリ」と「財布」を混同しない。カテゴリは支出の **種類** で、財布は支出を **どこから出すか** の管理単位。

## 2. コードに表せない不変条件 / 慣習

### 2.1 アクターは元帳上で常に `共同`

- 認証済みであれば、元帳に書き込む `actor` は常に文字列 `共同`（実装: `app/features/ledger/register.ts`）。
- 個人立替の判定は元帳の actor 列ではなく、別シートまたは運用ルールで二層管理する。
- **理由**: 元帳を「家計に属する取引のフラットなログ」に単純化することで、ダッシュボード集計のクエリと AI プロンプトを単純に保てる。
- 詳細: `docs/spec/calculation.md` §3。

### 2.2 `shouldSettle` のデフォルトは `true`

- 新規取引レコード登録時、`shouldSettle` の既定値は `true`。
- **理由**: 家族の共有家計簿という用途上、ほぼすべての取引が精算対象。例外（精算しない取引）はその都度フラグを倒す運用のほうがミスが少ない。
- 詳細: `docs/spec/calculation.md` §2。

### 2.3 通常財布名は `YYYY-MM通常` の機械生成

- 通常財布の名前は AI には決めさせず、AI が返した `date` の年月から `${YYYY}-${MM}通常` を組み立てる（実装: `app/features/ledger/register.ts` の `toLedgerEntry`）。
- **理由**: AI 側に「今月の財布名は何か」を推測させる必要をなくし、月をまたぐ取引や月末取引でも決定論的に財布を確定できる。
- 詳細: `docs/spec/data-model.md`。

### 2.4 元帳は append-only

- 既存レコードは原則修正しない。キャンセルは「赤黒処理」（マイナス額の新規レコード追加）で表現する。
- **理由**: スプレッドシートを直接編集する運用上、過去レコードの修正は変更履歴が辿りにくく、集計の整合が取れなくなる。

### 2.5 リレーションは文字列マッチング

- 財布名・カテゴリ名・アクター名は ID ではなく **人間可読の一意な文字列** で参照する。
- スプレッドシート上で人が直接編集する運用を前提とするため。表記揺れによるマッチング漏れは「マスタ未登録扱い」として明示的に検知する設計で許容する。

## 3. ドメイン境界（`app/domain/` のサブディレクトリ）

| ディレクトリ | 役割 |
|-------------|-----|
| `app/domain/ledger/` | 元帳エントリ、パース済みエントリ、レシートパーサーの IF |
| `app/domain/budget/` | 予算・カテゴリ・財布の集計ロジック |
| `app/domain/line/` | LINE クライアント IF、抽出済みメッセージ型、Webhook イベント |
| `app/domain/storage/` | スプレッドシート読み書きの IF、シート名定数 |

### AppError 階層の意図

`app/domain/errors.ts` で定義する派生例外（`LineApiError`, `GeminiApiError`, `ValidationError` 等）は、**呼び出し元での回復方針** を分けるためのもの。「どうユーザーに見せるか」「リトライ可能か」を `instanceof` で分岐できる粒度で切る。新規例外を足す前に、既存階層で足りないかを必ず確認する。
