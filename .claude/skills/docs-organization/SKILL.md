ドキュメントを書く・直す・整理する作業の前に読む。「この情報をどこに置くか」「Skill の description をどう書くか」「何を書かないか」を統一する。

---

## 1. 四層構造

| 層 | 役割 | エージェントの使い方 | 主な書き手 |
|----|------|------------------|-----------|
| `CLAUDE.md` | エージェントの入口。第一原則だけを置く | セッション開始時に自動で読まれる | 人間 |
| `.claude/skills/` | 規約・ドメイン知識・参照先マップ | description で発火、本文を読む | 人間 / エージェント |
| `docs/` | 過去の決定 + コードに表せない状態 | Skill から誘導されて辿る | 人間 / エージェント |
| `README.md` | セットアップ・dev起動・運用ガイド | 開発環境に触る作業時に参照 | 人間 / エージェント |

---

## 2. 何をどこに置くかの判断軸

新しい情報を書く前に、以下を順に問う。

1. **コードで表現できるか？** → コードで表現する。ドキュメントには書かない。
2. **エージェントが探索の起点にすべき原則か？** → `CLAUDE.md`（30 行以内）
3. **規約・ドメイン用語・対象ファイル名で発火させたい索引か？** → Skill 本文（要約 + 参照先）
4. **過去の決定 / コードに表せない暗黙ルール / 仕様の詳細か？** → `docs/`
5. **実行コマンド・環境変数・運用手順か？** → `README.md`

迷ったときは「**コードと一緒に変わるか**」を考える。コードと一緒に変わる情報はドキュメント化しない（コード自身が情報源）。コードを変えても変わらない決定や慣習だけがドキュメントに残る価値を持つ。

## 3. CLAUDE.md の書き方

### 書くもの

1. プロダクトの一行説明
2. 言語ポリシー（日本語統一など）
3. 第一原則（Skill 探索）
4. ディレクトリ役割の 3〜4 行サマリ
5. 振る舞いの制約（破壊的 git 操作の禁止、`--no-verify` 禁止など）

### 書かないもの

- Skill 一覧の列挙（Skill が増えるたび肥大化する。発火は description に任せる）
- 規約の中身
- ドメイン用語の定義
- セットアップコマンド・実行コマンド
- 技術選定の理由

### 目安

30 行以内。これを超えるなら、内容を Skill / docs / README に逃がす。

---

## 4. Skill の書き方

### 4.1 description の構造

Skill の description は **Claude が「このタスクで発火させるか」を判定する根拠** になる。説明書ではなくトリガとして書く。

**構造**: `[いつ使うか（ユースケース前置き）] → [何を提供するか] → [トリガキーワード列挙]`

仕様: description と when_to_use の合計は 1,536 文字で短縮されるため、それ以内に収める。

良い例（`.claude/skills/coding-standards/SKILL.md`）:

> app/配下のコード実装・レビュー・リファクタ時に使う。kakeibotのアーキテクチャ規範とコーディング規約を提供する。レイヤ越境ルール（domain/features/infra/routes）、features層のResult型返却（throw禁止）、valibot駆動の型導出（InferOutput / as Type禁止）、AppError階層によるエラーラップ、enum / any禁止、Cloudflare Workers（workerd）互換、AIへの推測委譲の境界。

NG パターン:

- 「常に / 必ず参照」だけ書く — 発火条件にならない。具体的な作業局面を書く。
- 何を提供するか書かずキーワードだけ並べる — 中身がわからず、誤発火する。
- ユースケースを最後に置く — 仕様の「主要ユースケースを前置き」に反する。
- 英語の TRIGGER / SKIP フォーマットを使う — kakeibot は日本語統一の言語ポリシーに従う。

### 4.2 本文の構造

- 章立ては「規約 / 不変条件 / Why の要約 / 参照先」を分ける。
- **コードを写経しない**。コードが変わるたびに乖離する。代わりに「規約 + 該当パスへの 1 行参照」を書く。
- **docs と二重記述しない**。詳細は docs を Single Source にし、Skill は要約 + 「詳細は `docs/...` 参照」だけ書く。
- TRIGGER / SKIP の明示的記法は使わない。description のキーワードで自然発火させる。

長さの目安: 100〜400 行。これを超えるなら章を分割するか、docs に逃がす。

### 4.3 ファイル配置

- 1 Skill = 1 ディレクトリ: `.claude/skills/<skill-name>/SKILL.md`
- 名前はハイフン区切りの英語（既存: `coding-standards`, `domain-knowledge`, `design-system`, `testing`, `docs-organization`）。

## 5. docs/ の書き方

### 書くもの

- **過去の決定**（spec / architecture）: 仕様、レイヤ構成、エラー伝搬、外部 API の前提
- **コードに表せない状態**: 暗黙の不変条件、運用上の慣習、システム俯瞰図（C4 等）

各決定は「**決定 / 背景 / 棄却した選択肢**」の 3 項で短く書くと、後で読み手が再評価できる。

### 書かないもの

- 実行コマンド（→ README）
- ドメイン用語の定義（→ `domain-knowledge` Skill 等）
- UI 規約（→ `design-system` Skill）
- バージョン番号・依存ライブラリ一覧（→ `package.json`）
- 個別ファイルパス・関数名のコピー（C4 図のような責務マップ目的は許容）
- 「YYYY 年 MM 月時点で」のような時間に依存する表現
- 「未実行」「TODO」「あとで」など作業バッファ的記述

### 構成

- `docs/spec/` プロダクト仕様（要件・データモデル・計算ロジック）
- `docs/architecture/` アーキテクチャ（横断的決定、技術選定、C4）
- `docs/README.md` 索引

### Why の集約方針

ADR ディレクトリは作らない。横断的な決定は `docs/architecture/overview.md` に「決定 / 背景 / 棄却した選択肢」の 3 項でまとめる。仕様レベルの決定は `docs/spec/calculation.md` などのドメインに近い場所に置く。

---

## 6. README.md の書き方

### 書くもの

- 前提条件・依存インストール
- 環境変数とその意味
- ローカル開発（dev サーバー起動、モック切り替えの組み合わせ）
- テスト / 型 / Lint の実行コマンド
- デプロイ手順

人間とエージェントの共用ガイド。「開発環境に触る作業のとき」エージェントもここを参照する。

### 書かないもの

- 規約・設計判断の Why（→ Skill / docs）
- ドメイン用語の定義（→ Skill）
- UI 規約（→ Skill）

---

## 7. 主要なアンチパターン

1. **CLAUDE.md に Skill 一覧を列挙する** — Skill 増加のたびに肥大化する。発火は description に任せる。
2. **Skill にコードを写経する** — コード変更で必ず乖離する。Skill には「ルールと Why の要約 + 参照先」のみ。
3. **Skill と docs に同じ Why を二重記述する** — Skill は要約 + docs への 1 行参照に徹し、本文は docs を Single Source にする。
4. **docs/ に運用手順を書く** — 運用手順は README の責務。docs は「過去の決定」と「コードに表せない状態」だけを持つ。
5. **README にドメイン定義や設計判断の Why を書く** — README は運用情報に純化。Why は Skill / docs。
6. **description に「常に / 必ず参照」とだけ書く** — 発火条件として弱い。具体的な作業局面とトリガ語を書く。
7. **「YYYY 年 MM 月時点で」「バージョン X 系」をドキュメントに書く** — 時間 / 依存に縛られた記述は腐る。Why だけ書き、版情報は `package.json` に任せる。
8. **新規 Skill を作るとき類似プラグインとの役割分担を曖昧にする** — 起動条件が衝突して、kakeibot の確定 UI 言語に汎用 Tone が侵食する等の事故が起こる。本文内で「どちらをいつ使うか」を明記する。

## 8. 新設 / 改修時のチェックリスト

ドキュメント / Skill を追加・改修したら、commit 前に確認する。

- [ ] 情報を置く層は §2 の判断軸で正しいか？
- [ ] CLAUDE.md は 30 行以内に収まっているか？
- [ ] Skill の description は「ユースケース → 提供内容 → トリガキーワード」の順か？
- [ ] Skill 本文がコードや docs と二重記述になっていないか？
- [ ] docs に実行コマンド・年号・バージョン番号・「未実行」記述が紛れていないか？
- [ ] README にドメイン定義や設計判断の Why が紛れていないか？
- [ ] 削除した旧ファイルへのリンクが残っていないか？（`grep -rn '旧ファイル名'`）
- [ ] 新規 Markdown を増やしていないか？増やすなら層と責務が既存と被っていないか？

## 9. 検証シナリオ

ドキュメント情報設計が機能しているかは、以下を Claude Code に投げて確認する。

- **実装タスク**: 「新しい予算カテゴリを追加する」 → `coding-standards` / `domain-knowledge` が発火、必要に応じて `docs/spec/data-model.md` に辿る。
- **UI タスク**: 「新しいダッシュボードカードを作る」 → `design-system` が発火。`frontend-design` プラグインを併用しても最後は `design-system` で締まる。
- **テスト追加**: 「features/budget/manage に新規テストを足す」 → `testing` が発火、実行コマンドは README に辿り着く。
- **仕様確認**: 「actor が共同に統一されているのはなぜ？」 → `domain-knowledge` → `docs/spec/calculation.md` の順に辿る。
- **dev 起動**: 「ローカルで webhook をモックで叩きたい」 → README の該当節に辿り着く。

辿れない場合の対処は **「Skill の description を調整する」** が基本。CLAUDE.md に情報を足して解決しようとしない（CLAUDE.md は薄く保つ）。
